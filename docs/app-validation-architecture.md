# App Validation Architecture: From Prototype to Production

## Problem Statement

The CMS Kill the Clipboard initiative has 12 early adopters and 71+ pledgees — health apps committed to supporting SMART Health Link QR codes. Healthcare providers want to verify that a patient's health app is on this approved list **before** scanning sensitive health data.

However, SMART Health Link QR codes don't identify the source app. We need a separate "app identity" verification step.

**The security challenge:** Any static identifier (app name, ID string) can be trivially copied and spoofed. A production system must cryptographically prove the QR code was generated by the actual approved app.

---

## Phase 1: Static Approved List (Current Prototype)

**Status:** Implemented and deployed

### How It Works
- App identity QR codes contain static JSON: `{"type":"cms-htei","name":"Apple","appId":"apple"}`
- Scanner checks `appId` against a hardcoded list of 83 CMS-approved apps
- Admin toggle enables/disables the two-step flow per organization

### Limitations
- Anyone can generate a valid app identity QR code — no cryptographic proof
- Approved list is hardcoded in server.js — requires redeployment to update
- No revocation capability

### Value
- Demonstrates the two-step scan flow concept
- Validates the UX with real users
- Proves the technical integration works

---

## Phase 2: Dynamic Directory API

**Status:** Proposed near-term improvement

### Changes from Phase 1
- Replace the hardcoded `APPROVED_APPS` array with a live API call to a centrally managed CMS directory
- The directory is maintained by CMS and updated as apps join or leave the program
- Scanner calls the directory at verification time rather than checking a local list

### Architecture
```
Patient's App                Scanner                 CMS Directory API
    |                          |                          |
    | [shows static QR]        |                          |
    | ──── QR code ──────>     |                          |
    |                          | GET /api/apps/{appId} ──>|
    |                          | <── { approved, tier } ──|
    |                          |                          |
    |                          | [show result]            |
```

### Directory API Endpoints
```
GET  /api/v1/apps                    # List all approved apps
GET  /api/v1/apps/{appId}            # Check single app status
GET  /api/v1/apps?tier=early-adopter # Filter by tier
```

### Response Format
```json
{
  "appId": "apple",
  "name": "Apple",
  "tier": "early-adopter",
  "status": "approved",
  "approvedDate": "2024-06-15",
  "organizationUrl": "https://www.apple.com/health/"
}
```

### What This Solves
- Centrally managed list — no redeployment needed when apps are added/removed
- CMS controls the source of truth
- Apps can be suspended or revoked in real time

### What It Doesn't Solve
- Still uses static QR codes that can be copied/spoofed
- No proof the QR was generated by the actual app

## Phase 3: Signed App Attestation (Production)

**Status:** Proposed production architecture

This is the architecture that prevents spoofing. It requires apps to cryptographically sign their identity QR codes using keys issued by CMS and published in the National Provider Directory.

### Key Concepts

**Certificate issuance:** When an app is approved through the Kill the Clipboard program, CMS issues them a signing key pair. The app keeps the private key; CMS publishes the public key in the National Provider Directory.

**Dynamic QR codes:** Instead of static JSON, the app generates a signed, timestamped token each time a patient needs to verify their app. The token expires after a few minutes.

**Signature verification:** The scanner verifies the signature against the app's public key from the directory. If the signature is valid and the token hasn't expired, the app is confirmed as legitimate.

### QR Code Format: JWS Compact Serialization

The app identity QR code contains a JWS (JSON Web Signature) string:

```
eyJhbGciOiJFUzI1NiJ9.eyJ0eXBlIjoiY21zLWh0ZWkiLCJhcHBJZCI6ImFwcGxlIiwiaWF0IjoxNzA5NTcxMjM0LCJleHAiOjE3MDk1NzE1MzQsIm5vbmNlIjoiYTNmOGMyIn0.MEUCIQDx...
```

**Header:**
```json
{
  "alg": "ES256",
  "kid": "apple-2024-001"
}
```

**Payload:**
```json
{
  "type": "cms-htei",
  "appId": "apple",
  "iat": 1709571234,
  "exp": 1709571534,
  "nonce": "a3f8c2e9b7d1"
}
```

| Field   | Purpose                                      |
|---------|----------------------------------------------|
| `type`  | Identifies this as a CMS HTEI app attestation |
| `appId` | Unique app identifier in the directory        |
| `iat`   | Issued-at timestamp (Unix seconds)            |
| `exp`   | Expiration (typically 5 minutes after `iat`)  |
| `nonce` | Random one-time value to prevent replay       |

### Verification Flow

```
Patient's App                Scanner                 CMS National Provider Directory
    |                          |                              |
    | [generate JWS token      |                              |
    |  signed with private key]|                              |
    |                          |                              |
    | ──── JWS QR code ──────> |                              |
    |                          |                              |
    |                          | 1. Decode JWS header         |
    |                          |    → extract appId, kid      |
    |                          |                              |
    |                          | 2. GET /apps/{appId}/jwks ──>|
    |                          |    <── public key(s) ────────|
    |                          |                              |
    |                          | 3. Verify JWS signature      |
    |                          |    using public key           |
    |                          |                              |
    |                          | 4. Check exp > now           |
    |                          |    (token not expired)        |
    |                          |                              |
    |                          | 5. Check nonce not reused    |
    |                          |    (replay protection)        |
    |                          |                              |
    |                          | [✓ Approved — show green]    |
    |                          |                              |
    | [show health QR]         |                              |
    | ──── SHL QR code ──────> |                              |
    |                          | [process health data]        |
```

### Scanner-Side Verification Steps

```
1. Read QR code
2. Detect JWS format (three base64url segments separated by dots)
3. Decode header → get `alg` and `kid`
4. Decode payload → get `appId`, `exp`, `nonce`
5. Fetch public key:
   GET https://directory.cms.gov/api/v1/apps/{appId}/jwks
   → Cache the JWKS for a configurable period (e.g., 1 hour)
6. Verify signature using the matching key (by `kid`)
7. Validate claims:
   - `exp` > current time (not expired)
   - `iat` < current time (not issued in the future)
   - `iat` > current time - 10 minutes (not too old)
   - `nonce` not in recent nonce cache (not replayed)
8. If all pass → app is verified
```

### National Provider Directory Endpoints

```
# App JWKS endpoint (public keys for signature verification)
GET /api/v1/apps/{appId}/jwks

# App metadata
GET /api/v1/apps/{appId}

# Full directory listing
GET /api/v1/apps

# Revocation check
GET /api/v1/apps/{appId}/status
```

**JWKS Response Format:**
```json
{
  "keys": [
    {
      "kty": "EC",
      "crv": "P-256",
      "kid": "apple-2024-001",
      "use": "sig",
      "x": "f83OJ3D2xF1Bg8vub9tLe1gHMzV76e8Tus9uPHvRVEU",
      "y": "x_FEzRu9m36HLN_tue659LNpXW6pCyStikYjKIWI5a0"
    }
  ]
}
```

### Why This Can't Be Spoofed

| Attack Vector                | Defense                                              |
|------------------------------|------------------------------------------------------|
| Copy someone else's QR code  | Token expires in ~5 minutes                          |
| Generate fake QR with format | Cannot sign without the app's private key            |
| Replay a captured QR code    | Nonce + expiration prevent reuse                     |
| Fake app claims to be Apple  | Signature verification fails without Apple's key     |
| Compromise one app's key     | CMS revokes that app's certificate in the directory  |
| Man-in-the-middle the JWKS   | HTTPS + certificate pinning on directory domain      |
| Create a fake directory      | Scanner is configured to trust only the CMS domain   |

### Key Management

**For apps:**
- CMS provides a developer portal where approved apps generate their signing keys
- Apps store their private key securely (e.g., iOS Keychain, Android Keystore)
- Keys can be rotated through the portal — old keys remain valid until their expiration

**For CMS:**
- The National Provider Directory publishes each app's JWKS endpoint
- CMS can revoke keys immediately by removing them from the JWKS
- Key rotation is supported — multiple active keys per app via the `kid` field
- Audit logging tracks all key operations

### Algorithm Choice

**ES256 (ECDSA with P-256 and SHA-256)** is recommended because:
- Same algorithm used by SMART Health Cards — familiar to the ecosystem
- Compact signatures (~88 bytes) keep QR codes small and scannable
- Widely supported in mobile platforms (iOS CryptoKit, Android KeyStore)
- NIST-approved, suitable for government systems

### Standards Alignment

This design intentionally mirrors existing standards in the SMART Health ecosystem:

| Standard              | How We Align                                               |
|-----------------------|------------------------------------------------------------|
| SMART Health Cards    | Same JWS format, same ES256 algorithm, same JWKS discovery |
| SMART on FHIR         | JWKS-based key publication and verification                |
| OAuth 2.0 / OpenID    | Standard JWT claims (iat, exp, nonce)                      |
| W3C DID               | Could extend to DID-based identity in the future           |

## Implementation Phases Summary

| Phase | Verification Method      | Spoofable? | Directory Required? | App Changes? |
|-------|--------------------------|------------|---------------------|--------------|
| 1     | Static hardcoded list    | Yes        | No                  | No           |
| 2     | Dynamic CMS API lookup   | Yes        | Yes (basic)         | No           |
| 3     | JWS signed attestation   | No         | Yes (with JWKS)     | Yes          |

### Migration Path

The QR code format is designed to be backwards-compatible:

- **Phase 1 QR codes** start with `{` (raw JSON)
- **Phase 3 QR codes** start with `eyJ` (base64url-encoded JWS)
- The scanner can detect the format and handle both during the transition period
- Apps can adopt JWS signing at their own pace while the static format continues to work

## Open Questions

1. **Key escrow:** Should CMS hold a backup of app signing keys, or should apps be solely responsible?
2. **Offline verification:** Should scanners cache JWKS for offline use in areas with poor connectivity?
3. **Multi-platform apps:** Does an app with iOS and Android versions get one key pair or separate ones?
4. **Attestation scope:** Should the signed token include device-level attestation (e.g., iOS App Attest, Android Play Integrity)?
5. **Directory SLA:** What uptime and latency guarantees does the directory need for real-time verification?
6. **Transition timeline:** How long should the Phase 1 static format be accepted alongside Phase 3 JWS?
